<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer's -update flag → test fixture files | A test-time write to testdata/ overwrites version-controlled fixtures. The threat is: accidental capture during normal CI (would corrupt fixtures), or a transient upstream outage producing a degenerate body (would corrupt fixtures). |
| live upstream → fixture sanity check | Hostile / outage upstream body must be rejected before it can clobber the committed fixture. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-Tampering-FixtureClobber | Tampering (test infrastructure) | update_fixtures_test.go | mitigate | **Double gate**: `//go:build integration` (compile-time) + `OPENHOLIDAYS_LIVE=1` env var (run-time). Neither alone is sufficient — both must be true for any live HTTP request OR fixture write to occur. Atomic write via `os.CreateTemp` + `os.Rename` ensures no partial files on SIGINT. `nonEmptyJSONArray` sanity check rejects upstream outage bodies before overwrite. Non-200 responses are rejected via `require.Equalf` on StatusCode before any write logic runs. |
| T-3-DoS-OverSize | DoS | readAll helper | mitigate | `io.LimitReader(r, 11<<20)` cap on live response read — even hostile streaming upstream cannot OOM the fixture refresher. |
| T-3-Tampering-EmptyBody | Tampering | nonEmptyJSONArray sanity | mitigate | Empty array, malformed JSON, or any non-JSON content fails the sanity check and ABORTS the overwrite. |
| T-3-DoS-CIAccidentalRun | DoS / cost | n/a | mitigate | Build tag excludes the file from normal `go test ./...` runs; CI workflows that DO want drift detection set both `-tags=integration` and `OPENHOLIDAYS_LIVE=1` explicitly. There is no way to accidentally hit the live API from a default test invocation. |
</threat_model>
