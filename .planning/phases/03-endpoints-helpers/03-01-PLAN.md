<objective>
Extract the shared HTTP-and-decode pipeline from `countries.go` into a new generic helper `doJSONGet[T any]` in `request.go`. Retrofit `Countries` to the uniform `(ctx, CountriesRequest)` shape that every Phase 3 endpoint follows. Move `buildAPIError`, `parseAPIMessage`, `maxResponseBytes`, and `apiErrorBodyCap` from `countries.go` to `request.go`. Refresh `testdata/countries.json` against the live API.

Purpose: Plans 2-5 add four new endpoints that ALL share the Phase 2 D-41..D-45 + D-24 pipeline. Without `doJSONGet[T]`, every endpoint would re-implement the pipeline and drift over time. Plan 1 makes the canonical pipeline a single function so Plans 2-5 each shrink to ~25 lines (validate → build query → `doJSONGet` → optional post-decode validate → return).

This plan is a **foundation plan**: it does NOT itself satisfy a roadmap requirement ID (no `ENDPT-*` / `HELP-*` / `TEST-*` row maps to "lift a pipeline into a generic helper"). The `requirements` frontmatter is therefore an empty array. Per-endpoint TEST-01 coverage lives in Plans 2-5 against their own endpoints; the Countries retrofit re-uses the existing Phase 2 Countries test suite.

Output: `request.go` + `request_test.go` (new); refactored `countries.go` + `countries_test.go`; refreshed `testdata/countries.json`. Per D-72, this plan gates all other endpoint plans.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → Client | Caller-supplied ctx and CountriesRequest cross into the SDK; nil ctx and empty-but-set LanguageIsoCode are the misuse cases. |
| Client → upstream | The OpenHolidays HTTP server is partially trusted (it's a public free API); a hostile response (oversize body, malformed JSON, 4xx error envelope with hostile content) must be capped + drained + sanitized before reaching the caller. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-DoS-OverSize | Denial of Service | request.go `doJSONGet[T]` | mitigate | `io.LimitedReader{N: maxResponseBytes}` caps decoder at 10 MiB; mid-truncation gate (`limited.N == 0`) and boundary gate (`decoder.More()`) both surface `ErrResponseTooLarge`. The drain in the defer is also `io.LimitReader`-capped at `maxResponseBytes+1`. Inherited from Phase 2 D-24/D-25 verbatim. |
| T-3-InfoDisc-APIErrorBody | Information Disclosure | request.go `buildAPIError` | mitigate | `io.LimitReader(resp.Body, apiErrorBodyCap)` caps `APIError.Body` at 4 KiB so a hostile multi-MB error body cannot bloat operator logs. Inherited from Phase 1 D-17 verbatim. |
| T-3-DoS-NilCtx | Denial of Service (caller-induced) | request.go `doJSONGet[T]` | mitigate | Defensive nil-ctx guard returns `"openholidays: nil context"` before any network operation; covered by unit test. |
| T-3-Tampering-EmptyBody | Tampering | request.go `doJSONGet[T]` decoder | mitigate | 200 + zero-byte body returns `ErrEmptyResponse` wrap so callers can branch on `errors.Is(err, ErrEmptyResponse)`. Inherited from Phase 2 D-46 / Plan 02-03. |
</threat_model>
