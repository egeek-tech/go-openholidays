<objective>
Document how consumers verify the SLSA build-provenance attestation on released `ohcli` binaries, in two places: a new README section and a tightened row in the release runbook's §3 verification matrix.

Purpose: The released archives are signed via GitHub's keyless (Sigstore/Fulcio) attestation flow, but nothing in the repo tells a consumer how to verify them — and the common mistakes (verifying the unpacked binary, or `checksums.txt`, or a locally-built binary) all return a confusing `HTTP 404: Not Found`. This documents the verified-correct recipe and explicitly warns against those three traps.

<verified_facts>
- The release attestation is produced by `actions/attest-build-provenance@v4.1.0` in `.github/workflows/release-please.yml` with input `subject-checksums: ./dist/checksums.txt`.
- The attested SUBJECTS are the six release ARCHIVES listed in `checksums.txt` (`ohcli_X.Y.Z_OS_ARCH.tar.gz` for linux/darwin, `.zip` for windows) — NOT the extracted `ohcli` binary, and NOT `checksums.txt` itself.
- `gh attestation verify` requires gh ≥ ~2.49 (verified working on gh 2.93.0).
- VERIFIED WORKING (exit 0): `gh attestation verify ohcli_0.5.0_linux_amd64.tar.gz --repo egeek-tech/go-openholidays`.
- VERIFIED FAILING (HTTP 404: Not Found): verifying the unpacked `./ohcli` binary (different sha256, not a subject).
- VERIFIED FAILING (HTTP 404: Not Found): verifying `checksums.txt` (it lists the subjects, it is not itself one).
- A locally-built binary (`go install …/cmd/ohcli@latest` or `go build`) is NEVER attested — only release-uploaded archives are. A 404 on a local build is expected, not a bug.
- Provenance chain on success: predicateType `https://slsa.dev/provenance/v1`; source repo `https://github.com/egeek-tech/go-openholidays`; source ref `refs/heads/master` (the workflow's triggering push ref, NOT the tag); build workflow `…/.github/workflows/release-please.yml@refs/heads/master`; OIDC issuer `https://token.actions.githubusercontent.com`.
- HARDENED-VERIFY GOTCHA: because the signing ref is `refs/heads/master`, `--source-ref refs/tags/vX.Y.Z` would FAIL. The correct hardened form pins the workflow: `--signer-workflow egeek-tech/go-openholidays/.github/workflows/release-please.yml`.
- `--repo egeek-tech/go-openholidays` is the preferred form; `--owner egeek-tech` also works.
</verified_facts>
